 # B2B 管理后台 · 方法论文档
 
 > 面向 B 端系统的产品需求、页面布局、交互规范与快速原型方法论。
 > 源于内部系统的实际项目迭代。
 
 ## 背景
 
 B 端管理后台的特点是：功能重复度高、数据密集、用户对效率和信息密度敏感。但每个新项目往往从零开始画原型、定布局、写模板，导致大量重复劳动和前后不一致。
 
 这套方法论的目标是：**把 B 端后台的设计决策成本压缩到最低**，让 PM 和开发能复用已有的模式，把精力放在业务逻辑而非页面样式上。
 
 ## 结构
 
 按工作流分为四个层次，从抽象到具体：
 
 ### 1. PRD 模板 → `references/prd-template.md`
 
 写 PRD 的骨架结构：问题陈述 → 解决方案 → 用户故事。覆盖了 B 端系统常见的 feature area 分类方式，拿到需求后照着填空即可。
 
 ### 2. 页面布局规范 → `references/layout-patterns.md`
 
 定义了三种标准页面类型及其组件构成：
 - **看板页**：统计卡片行 + 重点关注表 + 完整数据表
 - **数据列表页**：筛选栏 + 表格 + 分页
 - **表单/管理页**：内联编辑 / 弹窗 / 全页表单
 
 同时规定了导航分组按使用频率排列（而非按后端模块）、响应式断点等。
 
 ### 3. 交互设计原则 → `references/interaction-patterns.md`
 
 11 条从真实项目中提炼的原则，覆盖：
 - 数据密度优先、渐进式披露等顶层策略（#1-#6）
 - 快捷筛选按钮组、管理页统计行、弹窗数据能力等实战模式（#7-#11）
 - 每一条都附带 Why（为什么）和 How（怎么做），而非空洞的建议
 
 ### 4. 可运行的种子页面 → `assets/template.html`
 
 一个可直接在浏览器打开的单 HTML 文件，内置：
 - 侧边栏导航（含页面切换 JS）
 - 统计卡片、图表卡片、数据表格
 - 弹窗、表单、筛选、分页、状态徽标
 - Chart.js 折线图示例
 
 替换品牌色、logo 和导航项后即可作为项目起点。
 
 ## 使用方式
 
 对于 **PM**：建议先读 PRD 模板和布局规范，做功能划分时心里有底。
 对于 **前端开发**：种子页面开箱即用，样式体系（CSS 变量 + 组件化 class）可以直接沿用或转成组件库代码。
 对于 **想掌握这套方法的人**：从交互设计原则开始读，理解每个决策的 tradeoff 之后，再去读布局规范和模板会更顺。
 
 ## 产生背景
 
 本方法论形成于抖音搜一搜、抖音关键词排名监测后台的真实开发过程。涵盖：
 - 数据模型从扁平列表 → 一对多分配记录的演化
 - 看板图表类型、布局比例、筛选方式的反复打磨
 - 弹窗从静态展示 → 带时间筛选和导出功能的数据容器
 - 系统设置页从单列 → 左右两列布局
 - 表单组件在工具栏（inline）和弹窗（modal）中采用不同尺寸的原因
 
 每一轮迭代中暴露出来的共性问题都被提炼为交互设计原则当中的一条，最终沉淀为这份文档。
 
 ## 适用范围
 
 **适合**：
 - 运营后台、数据看板、管理面板、监测工具
 - 内部系统、B2B SaaS、审核/工单/报表类页面
 
 **不适合**：
 - C 端营销页、品牌官网、内容型站点
 - 需要大量动效和叙事节奏的交互式页面
 
 ## 说明
 
 这份文档不依赖任何框架或工具链，所有文件均为纯文本 / Markdown / HTML，可在任何环境下阅读和编辑。种子页面使用原生 HTML + CSS + Chart.js，零构建步骤，适合快速原型验证。
